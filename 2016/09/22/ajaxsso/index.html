流程

jsServer登录

Title: App Server1登录 Browser->JsServer: 访问首页P JsServer->Browser: 返回不需要权限访问的首页 Browser->Cas: 浏览器初始化首页，并请求login ticket(a) Cas->Browser: 返回LT(b) User->Browser: 输入用户名密码并登录 Browser->Cas: jsonp或iframe提交登录表单(c) Cas->Browser: 登录并返回TGC， ST(d) Browser->JsServer: (e)用ST请求JsServer登录 JsServer->Browser: 登录成功并返回权限列表(f)(1) Browser->User: 根据权限列表，初始化界面

(a): jsonp 请求 url: https://cashost/fetchlt
调用示例:

(b) 返回login ticket
data 格式 {"code":0, "lt": login-ticket}
code:

  • 0 成功

lt: login ticket在提交登录表单时，一起提交

(c) 提交登录表单
url: https://cashost/flogin?service=http://127.0.0.1:8090/cas/login
url中service参数可选, 见(d)

userName=10000&password=123456&loginTicket=LT-6-a5mzMcnnFTVi3odCAIBOODHlrmjLK5-simpletour.com&rememberMe=false
userName: 用户名
password: 密码，直接提交
loginTicket: 上一步返回的lt
rememberMe: 是否自动登录

(d) 验证登录信息并以cookie的形式返回TGC(Ticket granting ticket cookie)
若存在Service， 则返回JSON数据中包含ST(service ticket), 否则为空
返回示例：
{"code": 0, "msg": "登录成功", "st":"ST-2-KhDgXWulPePADUS51Ssl-simpletour.com", "service": "http://127.0.0.1:8090/cas/login"}
code:

  • 0 成功
  • 1 失败， 详细信息使用msg

若存在ST， 则继续后边的登录应用服务器的步骤

(e): ajax请求(貌似要用jsonp), JsServer可以替换为其他server，如portal, 根据(c)中的service决定

url参数
ticket: 上一步返回的service ticket
needUserInfo: 是否需要返回userInfo

  • true 是
  • false 否

(f): 返回登录结果

code:

  • 0 成功
  • 1 失败， 详细信息使用msg

AppServer 通过浏览器跳转登录(现有流程)

Title: App Server2 登录(GET, POST 通过浏览器跳转) Browser->AppServer2: get请求App Server2页面P AppServer2->Browser: redirect到Cas Browser->Cas: 请求登录 Cas->Browser: 发现已登录，返回redirect & ST Browser->AppServer2: 携带ST登录 AppServer2->Cas: 根据ST请求用户信息 Cas->AppServer2: 返回用户信息 AppServer2->Browser: redirect到登录前页面P Browser->AppServer2: 请求页面P

AppServer 通过Ajax登录，页面不跳转

Title: App Server2 登录(Ajax, 所有数据交换通过JSON完成) Browser->AppServer2: ajax get | post请求App Server2页面P(a) AppServer2->Browser: 返回JSON， Code=1 未登录(b) Browser->Cas: 携带Server id发起Ajax请求(c) Cas->Browser: 发现已登录，返回JSON Server id & ST(d) Browser->AppServer2: ajax 携带ST登录(e) AppServer2->Cas: 根据ST请求用户信息 Cas->AppServer2: 返回用户信息 AppServer2->Browser: 返回登录成功，可包含用户信息(f) Browser->AppServer2: 请求页面P

(a), (e): jsonp 请求中需要指明是否为jsonp请求(用作区分普通请求)(jsonp,还是ajax?)
针对ajax, x-requested-with 可以用作区分，但不保证所有浏览器可用，所以最好增加一个自定义header
eg: request-jsonp

(b): 返回数据格式(若用户已登录，则不会有此以及下面的步骤, 而是直接返回数据)
{"code": 1, "msg": "未登录"}
code:

  • 1 未登录(这里的未登录是指未登录AppServer2, 至于Cas是否登录，在下一步做验证)

(c): 去cas请求st jsonp请求
url: http://cashost/fetchst?service=http://127.0.0.1:8090/cas/login

(d): 返回
{"code":0, "msg"="", st="service-ticket", service="http://127.0.0.1:8090/cas/login"}
code:

  • 0 成功
  • 1 cas未登录， 需要客户端跳转或弹出登录页面
  • 2 未知错误，一般不会出现，客户端就处理为重新登录就好

st: service ticket
service: 请求中的service

(e): 在(d)成功的情况下
jsonp请求登录AppServer2

url参数
ticket: 上一步返回的service ticket
needUserInfo: 是否需要返回userInfo

  • true 是
  • false 否

(f): 返回登录结果

code:

  • 0 成功
  • 1 失败， 详细信息使用msg

所有Jsonp都使用callback

以上三种方式, 可以完成正常访问流程下的登录逻辑

问题：以上流程建立在假设所有数据的访问都在首页通过ajax(jsonp)请求完成

Browser中权限列表若保存在页面js中，则需要单独提供权限查询接口，否则页面关闭后再打开则不再有权限数据
如此，(1)处仅需返回登录成功，权限根据查询接口再次查询即可

对于不通过首页直接get数据的情况，若未登录，则会通过现有方式跳转到现在的登录界面
