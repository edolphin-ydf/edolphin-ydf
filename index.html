游戏服的邮件服务地址配置错误导致玩家卡住的问题分析

现象

游戏中的玩家卡住，无法进行操作

尝试断线重连， 大概率会卡住(实为1/3左右)

排查

查看游戏服日志，两台正常， 一台有报错，连接不上邮件服务器
还有重连网关的日志

查看网关日志，有和游戏服连接超时的错误

最初以为是两个问题，

  • 连接超时(因为网关和游戏服跨公网)
  • 配置错误

解决

修正游戏服配置的邮件服务器地址, 观察一段时间后发现，服务稳定了, 所以连接超时应该是与配置有关

分析

理论上， 邮件服连不上， 应该只会影响邮件系统才对， 为什么会导致玩家游戏卡住？

通过查看代码发现，使用grpc， 请求邮件接口的时候， context都是使用的context.Background(), 未使用超时配置(context.WithTimeout(context.Background())), 即连不上邮件服务器是不会有超时，会一直卡住

但是，所有邮件接口的请求都是在玩家协程， 每个玩家一个协程， 按理说应该只是卡住玩家协程才对，为什么会影响所有人?

如果一个玩家卡住， 则不会再消费消息，会导致玩家的消息缓冲队列满， 而网关转发到游戏服的新的消息， 无法再加入到卡住的玩家消息缓冲队列， 至此， 网关和游戏服连接卡住

进一步解决

游戏服调用邮件服务接口时， 要使用timeout

linux go too many open files问题记录

现象

服务器突然无法链接，所有用户都卡在登录界面，客户端确定为链接网关失败

排查

查看网关日志， 大量的 too many open files, 确定为连接数过多导致

ulimit -a 查看系统配置， open files一项为1024，问题根源找到

解决

ulimit -n 10240 修改最大支持打开文件为10240， 重启网关

cat /proc/$pid/limits 确认open files一项已被修改为10240

备注

每使用一台新服务器， 都应该查看ulimit -a各项参数最大值，并根据需求调整

Gw2上的只为65535, Gw3为另一云服务商， 各个云服务商的默认配置不同，需要注意

Mysql能在root环境下,通过root账户登陆,却不能在其他用户环境下通过root账户登陆解决办法

参考: https://askubuntu.com/questions/766334/cant-login-as-mysql-user-root-from-normal-user-account-in-ubuntu-16-04

思路就是

  • sudo su 切换到root账户
  • mysql连接到mysql
  • 删除旧root账户
  • 建立新root账户

具体流程:

  • sudo su
  • 使用linux的root账户连接 sudo mysql -u root
  • 删除root, DROP USER 'root'@'localhost';
  • 重建root, CREATE USER 'root'@'%' IDENTIFIED BY '';
  • 授权 GRANT ALL PRIVILEGES ON *.* TO 'root'@'%' WITH GRANT OPTION;
  • FLUSH PRIVILEGES;

退出mysql, 在不使用sudo的情况下 连接mysql

如果需要给重建的root设置密码

SET PASSWORD FOR 'root'@'%' = PASSWORD('new_password');

Debian下彻底删除mysql

参考文章：https://wangye.org/blog/archives/862/

  1. dpkg --get-selections | grep mysql 查看安装的与mysql相关的所有包
  2. sudo apt-get --purge remove mysql... 用上一步查询出来的结果 替换mysql… 依次删除每一个安装
  3. apt-get autoremove && apt-get autoclean
  4. rm /etc/mysql/ -R
  5. rm /var/lib/mysql/ -R

注意！！
以上， 会删除数据
