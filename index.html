2019-03-25

online-error-gem-data-type-is-0

现象

玩家切割原石时，游戏黑屏。重连后，一直卡在登录页面

排查

根据玩家id查询日志， 连续查了好几个玩家，共同点都是没有看到玩家切割原石的操作日志。当搜索ERROR日志时，发现在12:10左右，游戏服和网关断开过链接，但是没看出二者有和关联, 唯一的关系就是问题似乎都在这个时间点之后。

尝试自己切割原石，运气不错，一次就复现了问题。

查看日志，依然没有切割原石的操作日志，这样子就很奇怪了！！！

尝试排查代码，怎没也看不出问题。

再仔细看自己复现的时间，发现查到的日志都没有出问题那段时间的日志，想到只查了新服务器的日志，而忽略了旧服务器日志(想当然的认为，旧服务器已经被禁用掉了，却没想过会因为bug而导致旧服务器被重新启用)。

果断查询旧服务器日志，果然，旧服务器上该有的日志都有，找到现场，确认是切割原石，且返回和保存了GemData.Type为0的数据

再次查看代码，依然没发现为什么会出现错误的数据。

经某人提醒，是否和上次更新有关(上次更新，原石去掉了Type类型)? 再联想到，玩家连上了旧的服务器，由此确认问题：

  • 玩家在上次更新后，连上了新的服务器，生成的原石Type全为0
  • 因为未知的bug， 导致玩家在拥有了Type为0的原石后， 连上了旧的服务器
  • 旧服务器切割原石的逻辑中，生成的宝石， Type为直接使用原石的类型
  • 因此，玩家数据中出现了GemData.Type为0的数据

那么，为什么会连上旧的服务器呢？这个可是导致问题的根源！！

联想到12:10的断线，问题也就明朗了

  • 断线前，游戏服被禁用，但禁用是基于链接的，也就是说，链接断开，则禁用无效了
  • 重连后，是新的连接，禁用无效，导致玩家进入了旧的服务器，从而导致以上bug的发生

解决

找到问题根源之后，问题修复旧很简单了

  • 关闭旧服务器
  • 修复玩家数据

思考

旧服务器禁用机制原本是为了让玩家无感更新，结果因为开发中考虑不周，导致出现了此次的bug是问题根源所在，所以优化更新流程势在必行！

问题排查时，因为一直没有找到正确的日志，也导致浪费了很多时间, 排查问题时，不应先给自己设定某些不正确的前提条件，或者说，当未能发现问题是，可以第一时间将搜查范围扩大

linux go too many open files问题记录

现象

服务器突然无法链接，所有用户都卡在登录界面，客户端确定为链接网关失败

排查

查看网关日志， 大量的 too many open files, 确定为连接数过多导致

ulimit -a 查看系统配置， open files一项为1024，问题根源找到

解决

ulimit -n 10240 修改最大支持打开文件为10240， 重启网关

cat /proc/$pid/limits 确认open files一项已被修改为10240

备注

每使用一台新服务器， 都应该查看ulimit -a各项参数最大值，并根据需求调整

Gw2上的只为65535, Gw3为另一云服务商， 各个云服务商的默认配置不同，需要注意

Mysql能在root环境下,通过root账户登陆,却不能在其他用户环境下通过root账户登陆解决办法

参考: https://askubuntu.com/questions/766334/cant-login-as-mysql-user-root-from-normal-user-account-in-ubuntu-16-04

思路就是

  • sudo su 切换到root账户
  • mysql连接到mysql
  • 删除旧root账户
  • 建立新root账户

具体流程:

  • sudo su
  • 使用linux的root账户连接 sudo mysql -u root
  • 删除root, DROP USER 'root'@'localhost';
  • 重建root, CREATE USER 'root'@'%' IDENTIFIED BY '';
  • 授权 GRANT ALL PRIVILEGES ON *.* TO 'root'@'%' WITH GRANT OPTION;
  • FLUSH PRIVILEGES;

退出mysql, 在不使用sudo的情况下 连接mysql

如果需要给重建的root设置密码

SET PASSWORD FOR 'root'@'%' = PASSWORD('new_password');

Debian下彻底删除mysql

参考文章：https://wangye.org/blog/archives/862/

  1. dpkg --get-selections | grep mysql 查看安装的与mysql相关的所有包
  2. sudo apt-get --purge remove mysql... 用上一步查询出来的结果 替换mysql… 依次删除每一个安装
  3. apt-get autoremove && apt-get autoclean
  4. rm /etc/mysql/ -R
  5. rm /var/lib/mysql/ -R

注意！！
以上， 会删除数据
